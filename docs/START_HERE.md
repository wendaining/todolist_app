# 现在先做什么（先别慌）

你现在只需要做 3 件事：

1. 锁定 MVP 范围（不再加新功能）
2. 按规格做第一个可演示版本（Web）
3. 再决定技术栈细节和客户端扩展

## 当前 MVP（已确认）

1. 新建任务，勾选完成，完成历史保留
2. 任务支持优先级
3. 任务支持 DDL，并在到期前提醒
4. 多端云同步（使用 Token，不用账号密码）

## 推荐顺序（非常简化）

1. 先读 [SPEC-v0.1.md](./SPEC-v0.1.md)
2. 实现 Web 版最小闭环：新增 -> 完成 -> 同步 -> 提醒
3. 每完成一个功能，就更新一次规格文档
4. MVP 稳定后，再做桌面端/Android

## 今天就能开始的任务

1. 把数据模型和接口先定下来（见规格文档）
2. 写出最小 API（创建任务、更新状态、获取列表、同步）
3. 做一个最简单的 Web 页面验证流程

## 暂时不要做的事

1. 不要先做复杂 Agent 功能
2. 不要先追求多端同时开发
3. 不要先引入太多中间件

## 卡住时怎么做

1. 回到规格文档，把问题写成一句话
2. 只解决当前里程碑的问题
3. 每次只改一个小点，保证可验证

## 关于 Linux/WSL（最新决策）

结论：现在开始迁移到 Linux/WSL。

为什么现在迁移：
1. 你后续会接触后端部署、容器和脚本，Linux 环境更接近生产。
2. 越早统一环境，后面越少返工。

迁移原则：
1. 只迁移开发环境，不改业务需求。
2. 保持文档和接口不变，代码按里程碑推进。
3. 一次只做一件事：先跑通环境，再写功能。

执行顺序（简版）：
1. 安装并启用 WSL2（建议 Ubuntu）。
2. 在 WSL 内安装 Git、Node、包管理器和数据库工具。
3. 重新克隆项目并打开工作目录。
4. 跑通最小开发命令（先不写新功能）。
5. 记录差异和问题，再进入功能开发。
